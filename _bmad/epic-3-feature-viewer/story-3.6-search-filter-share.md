# Story 3.6: Search, filter & share

- **Epic:** Epic 3 — Realistic Feature Viewer ⭐
- **Status:** Draft
- **Priority:** P1
- **Depends on:** 3.2 (feature cards grid — provides the card + health-badge markup these filters act on), 3.3 (feature drill-down — provides the per-feature issue/node surfacing these filters narrow). Builds on the 3.1 served-SPA shell. Reads the confidence field from 1.3 and the health band from 2.4 when present, degrading gracefully when absent.
- **Estimate:** M

## Story
As a developer exploring an unfamiliar repo through the Sutra viewer, I want to search for any feature, endpoint, or node by name, narrow the view by health band, confidence threshold, and issue kind, and then share the exact filtered view I am looking at via a deep-link or a static export, so that I can hand a teammate a precise, reproducible slice of the graph instead of telling them to "open the viewer and scroll to roughly here."

## Context
Story 3.1 turned the static `view.html` dump into a served SPA (`src/viewer/server.ts` + `viewer/app.js`) that fetches `.sutra/graph.json` and renders the feature grid + detail panels via the shared helpers in `src/viewer/render-shared.ts`. Stories 3.2 and 3.3 put feature cards, the `feature.health.band` badge (from Story 2.4), and per-feature issue/node drill-down on screen. But the SPA is still a single scroll: there is no way to locate a specific node by name, to focus on (for example) only `red`-band features above a confidence threshold, or to point a colleague at one particular view. This breaks down the moment a real repo produces a real graph — `NOTES.md` records echo-ai at 638 nodes / 30 features and brain-api at 1,218 nodes / 18 features. Search, filter, and share is the last interaction gap before the viewer is genuinely browsable; ROADMAP.md Epic 3 line **3.6** ("Search, filter & share — find a feature/endpoint; filter by health/confidence; export a view") names exactly this.

This story completes the viewer UX and must respect Sutra's honesty contract (ROADMAP.md cross-cutting principle 1, *Never overstate*; `README.md` Claim Bounds; `BRIEF.md`). Candidate findings stay labelled candidate, low-confidence items stay visible (a filter hides them by user choice, never silently re-labels them), AI-derived fields keep their AI label, and a shared link reproduces a *view*, never a different *truth*. The graph itself is untouched — this is presentation and addressing only, so there is **no `graph.json` schema change and no `GRAPH_VERSION` bump**.

## Acceptance Criteria
1. The viewer SPA (`viewer/app.js`, served by `src/viewer/server.ts`) renders a search box that matches, case-insensitively and as a substring, against `SutraNode.name`, `SutraNode.id`, `SutraFeature.label`, and the endpoint/route strings carried in `SutraNode.name` for `type: "route" | "endpoint"`. As the user types, the visible feature cards and drill-down nodes live-filter. Matching is deterministic substring matching — no fuzzy/ML ranking that could reorder results nondeterministically across runs (ROADMAP principle 3, deterministic ids; principle 1, never overstate).
2. A health-band filter exposes one toggle per `HealthBand` value (`green | amber | red`, the type added in Story 2.4 in `src/types.ts`). Toggling a band shows/hides exactly the features whose `feature.health.band` matches, and never mutates the node/edge/feature counts shown in the SPA header. If `feature.health` is absent (a `version`-mismatched or pre-2.4 graph), band toggles degrade to an "unscored" bucket rather than throwing.
3. A confidence-threshold control (slider plus numeric readout) hides any node/issue whose `confidence` (the optional `0..1` field added on `SutraNode`/`SutraEdge`/`SutraIssue` in Story 1.3) is below the selected threshold. Bounds are `0..1`; elements with **no** `confidence` value (older graphs, un-annotated producers) are treated as "unknown" and are shown only when the threshold is at its minimum (`0`), with this rule stated in the UI so the user is never misled into thinking unknown == high confidence.
4. An issue-kind filter exposes one toggle per `IssueKind` value (`orphaned_endpoint | missing_handler | dangling_test_ref`, from `src/types.ts`) that actually appears in the loaded graph's `issues[]`. Selecting kinds restricts the view to features/nodes carrying at least one `SutraIssue` of a selected `kind`. Kinds defined in the type but absent from the current graph render no toggle (no dead, always-empty controls).
5. Filters compose with AND semantics: `search text` AND `active health bands` AND `confidence ≥ threshold` AND `(no issue-kind selected OR ≥1 matching issue kind)`. A live "showing X of Y features · N of M nodes" readout in the SPA always tells the user exactly how much is hidden.
6. A "Share this view" control serializes the current filter state (search string, active band set, confidence threshold, active issue-kind set) into the URL hash fragment (`location.hash`) and copies the full URL to the clipboard. Loading the viewer with that fragment restores the identical filter state on startup. The fragment is an addressing mechanism only — it carries no graph data and cannot alter the rendered graph, only what is filtered (ROADMAP principle 5, renderer is a leaf).
7. An "Export view" control produces a deterministic, self-contained static HTML file under `.sutra/` (e.g. `.sutra/view.<slug>.html`) that bakes in the current filter state, so a teammate without the running server or the deep-link can still open exactly this slice offline. The export reuses the existing static render path (`src/view.ts:renderView` via the shared `src/viewer/render-shared.ts` helpers) with the filter state pre-applied, and is **byte-stable** for identical graph input + identical filter state (regression-guardable).
8. Honesty is preserved across every filtered and exported view: candidate findings remain labelled candidate; the Story 3.1 "heuristic / candidate — review before acting" disclaimer (the `view.ts` `.disclaimer` block carried into the SPA) is present; any AI-derived field keeps its AI label; the confidence filter hides but never re-labels low-confidence items; and a filter that hides everything shows an explicit empty state ("0 of M nodes match — clear filters") rather than a blank page that reads as "nothing here."
9. All search/filter/share logic runs client-side in `viewer/app.js` with no new runtime network calls beyond the existing `GET /graph.json`, and the exported `.sutra/view.<slug>.html` is fully self-contained and opens offline — consistent with the local-first, single-user, no-Brain-runtime constraints in `BRIEF.md` and Story 3.1.

## Technical Approach
- **`viewer/app.js`** (from Story 3.1) — primary change. After it fetches and renders the graph via the `src/viewer/render-shared.ts` helpers, add a filter bar above the feature grid: (a) search input; (b) health-band toggle group built from `HealthBand` (`green|amber|red`) plus an "unscored" bucket for features lacking `health`; (c) confidence-threshold slider (`0..1`) with numeric readout; (d) issue-kind toggle group built from the **distinct** `IssueKind` values actually present in the fetched `graph.issues`; (e) the live "showing X of Y" readout; (f) "Share this view" and "Export view" buttons; (g) an empty-state element. The filter is applied by toggling visibility of already-rendered card/node DOM (produced server-side-shaped from `graph.json`); the client never recomputes graph data, preserving determinism and the leaf-renderer principle.
- **Filter-state codec — `src/viewer/viewState.ts` (NEW)** — a single pure, DOM-free `encodeViewState(state)` / `decodeViewState(str)` pair shared by the deep-link path and the static-export path so both yield identical results. Canonical key ordering; band and issue-kind sets serialized as sorted, deduped lists; confidence fixed to a stable precision (e.g. 2 decimals) so the same state always produces the same fragment. This module is unit-testable in isolation (no `window`/DOM).
- **Deep-link** — "Share this view" calls `encodeViewState`, sets `location.hash`, and copies `location.href`. On SPA load, if `location.hash` is non-empty, `decodeViewState` hydrates the initial filter state before the first filter pass. The hash is purely an addressing layer over the same graph the server returns from `GET /graph.json`.
- **Static export — `src/view.ts` + `src/viewer/render-shared.ts`** — extend `renderView` (or add a thin `renderFilteredView(graph, state)` beside it that reuses the same shared helpers) to accept an optional pre-applied filter state and emit only the matching cards/nodes, with filters reflected in the baked HTML. The export must stay byte-stable, so it reuses the same deterministic ordering already in the static renderer. Filename slug is derived deterministically from the encoded state via a new `slugifyViewState()` helper. **Note:** `src/util/ids.ts` today exports only `makeNodeId`, `toPosix`, `relPosix`, `httpTargetId` — there is **no** slug/hash helper — so add `slugifyViewState()` (NEW) either to `src/util/ids.ts` or alongside the codec in `src/viewer/viewState.ts`; it must be deterministic (stable hash/encoding, no `Date.now()`, no randomness).
- **`src/viewer/server.ts`** — the export is written to disk by the same path that wrote `.sutra/view.html`; if the SPA triggers export via an HTTP call, add one narrow `POST`/`GET` route on the existing localhost-only server that writes `.sutra/view.<slug>.html` and returns the path. Keep the path allowlist + `..`-traversal guard from Story 3.1; bind stays `127.0.0.1` only. (Alternatively the export can be a CLI affordance — pick one and note it; either way it reuses the same render path.)
- **Types / contract** — **no change to `src/types.ts` and no `GRAPH_VERSION` bump.** Filters read existing fields only: `SutraNode.name`/`id`/`type`, `SutraFeature.label`/`health.band` (2.4), `SutraNode`/`SutraIssue.confidence` (1.3), `SutraIssue.kind` (`IssueKind`). Per ROADMAP principle 4, a bump is reserved for breaking schema changes — there are none here.
- **No new checks** — `src/checks.ts` (`checkOrphanedEndpoints`, `checkMissingHandlers`, `checkDanglingTestRefs`) is unchanged; the issue-kind filter only *surfaces* existing `graph.issues`.
- **Honesty rules respected** — substring (not fuzzy/ML) search keeps results deterministic; the confidence filter hides without re-labelling; candidate/AI labels survive filtering and export; the "heuristic / candidate" disclaimer is carried into every view; deterministic slug/codec keep the export reproducible for `sutra diff`-style stability.

## Test Plan
New fixtures under `tests/fixtures/`:
- `tests/fixtures/filter-rich/.sutra/graph.json` (NEW, pre-built like the Story 3.1 `viewer` fixture) — a valid `SutraGraph` with features across all three `HealthBand` values, nodes/issues with varied `confidence` (including at least one element with **no** `confidence`), at least two distinct `IssueKind` values present in `issues[]`, and at least one candidate finding. Proves the full filter matrix has something to act on. (Add a `.gitignore` exception if `.sutra/` is globally ignored, matching Story 3.1's note.)
- `tests/fixtures/filter-empty/.sutra/graph.json` (NEW) — a minimal graph used to exercise the empty-state path.

New `describe` blocks in `tests/sutra.test.ts` (continuing the Section-numbering scheme `NOTES.md` describes):
- **`describe("viewState encode/decode")`** — round-trips representative states (search text with spaces/special chars; multiple active bands; fractional confidence threshold; a subset of issue kinds) and asserts `decodeViewState(encodeViewState(x))` deep-equals `x`. **Regression guard:** assert `encodeViewState` is byte-identical across two calls for the same state, and that `slugifyViewState` is stable (determinism, ROADMAP principle 3).
- **`describe("filtered static export")`** — render the filtered static view for `filter-rich` with a fixed state and assert: only matching features/nodes appear; band/confidence/issue-kind filters compose with AND; the "heuristic / candidate" disclaimer and candidate labels are present. **Regression guard:** generate the export twice with the same graph + state and assert byte-identical output and a stable `<slug>`.
- **`describe("honesty under filter/export")`** — assert the confidence filter hides a low-confidence element at a raised threshold but the same element retains its candidate/confidence labelling when shown; assert a feature with `health.band` "red" is not silently relabelled by any filter; assert AI-labelled fields (if present) keep their label in the export.
- **`describe("issue-kind toggles reflect present kinds")`** — for `filter-rich`, assert toggles are generated for exactly the distinct `IssueKind` values in its `issues[]` and not for kinds defined in the type but absent (no dead toggles).
- **`describe("empty-state")`** — with `filter-empty` (or a state that hides everything in `filter-rich`), assert the rendered output includes the empty-state markup so an all-hiding filter cannot produce a silently blank view.
- **Leaf/regression guard** — assert the new code paths do not alter the existing `renderView` (unfiltered) byte output, do not import `scanner.ts`/`checks.ts`/`features.ts` into the viewer, and that Sections 1–9 of the suite stay green (ROADMAP principles 5 + 7).

## Out of Scope
- Any change to `graph.json`'s schema or a `GRAPH_VERSION` bump — this is pure presentation/addressing over the existing contract.
- New structural checks in `src/checks.ts` or new `IssueKind` values — the issue-kind filter only surfaces what the scanner already emits.
- Server-side / hosted sharing (auth, upload, short-link service, remote hosting, `0.0.0.0` exposure) — forbidden by `BRIEF.md` hard constraints. Sharing here is a localhost deep-link fragment plus a local self-contained HTML export only.
- Fuzzy / ML / semantic search ranking — search is deterministic substring matching to honour the no-nondeterminism rule.
- Persisting a user's last-used filter across sessions (e.g. `localStorage`) — deferred to a later viewer-polish story.
- Live/watch re-scan and push-to-viewer — Story 3.5.
- Cross-repo map filtering (filtering across the ecosystem/cluster view) — Story 3.4; this story filters within a single graph.
- Defining the confidence model (Story 1.3) or the feature health/band (Story 2.4) — this story only *reads* those fields and degrades gracefully when absent.
