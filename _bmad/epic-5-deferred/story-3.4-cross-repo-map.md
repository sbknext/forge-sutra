# Story 3.4: Cross-repo ecosystem map

- **Epic:** Epic 3 — Realistic Feature Viewer
- **Status:** Draft
- **Priority:** P1
- **Depends on:** 3.1 (viewer app shell — the `sutra viewer` command, `src/viewer/server.ts:startViewerServer`, the zero-build `viewer/` SPA, `src/viewer/render-shared.ts`, and the `GET /graph.json` fresh-disk-read route); 1.4 (cross-repo linking — `src/link.ts:linkGraphs`, the `LinkResult`/`LinkedEdge`/`resolution` types in `src/types.ts`, `LINK_VERSION`, the `.sutra/link.json` artifact, and the `makeCrossRepoId(repo, nodeId)` → `<repo>::<relPath#symbol>` id convention in `src/util/ids.ts`); 2.2 (client↔server reconciliation — `SutraReconciliation`/`SutraReconItem`/`ReconStatus` on `SutraGraph`, and `GRAPH_VERSION = 1`).
- **Estimate:** L

## Story
As a developer surveying a multi-repo product (echo-ai ↔ brain-api), I want the viewer to render an **ecosystem map** — each repo as a cluster, with **confirmed** cross-repo edges drawn between clusters and each link annotated with its reconciliation status — so that I can see, honestly, how the product's services actually wire together and where a verified seam differs from a merely-suspected one.

## Context
ROADMAP.md Epic 3 names this story verbatim: "**3.4** Cross-repo map (ecosystem view: repos as clusters, confirmed cross-repo edges between them)". It is the final node on the minimum path to the named goal (`… → 3.1 → 3.2 → 3.3 → 3.4`), and the Definition of Done explicitly requires the viewer to "show the **cross-repo** echo-ai↔brain-api map". NOTES.md is the concrete motivation: echo-ai's entire `/api/*` + `/auth/*` surface is proxied to brain-api (54 candidate `orphaned_endpoint`s), and brain-api's own scan found 320 Express endpoints — the seam between those two repos is exactly what the single-repo viewer cannot show.

This story is a **viewer feature**: it does not produce cross-repo edges (Story 1.4 does, into `.sutra/link.json` as `LinkResult`) and it does not compute reconciliation status (Story 2.2 does, into each graph's `SutraGraph.reconciliation`). 3.4 consumes both and renders them as an ecosystem-level cluster graph inside the existing `viewer/` SPA from 3.1. One real plumbing gap must be closed here: the 3.1 shell only serves the **current** repo's `graph.json` via `GET /graph.json` (a fresh disk read on each request); it exposes no route for `.sutra/link.json` and no way to fetch a *second* repo's graph. So this story extends `src/viewer/server.ts` with two additive read-only routes (`GET /link.json` and a per-repo graph route) and leaves everything else 3.1 built untouched. The roadmap honesty principles are load-bearing: only `LinkedEdge`s with `resolution: "confirmed"` may render as solid links; `broken` and `unresolved` links must be visibly distinct and never promoted to look confirmed (ROADMAP principle 1, "Never overstate").

## Acceptance Criteria
1. The 3.1 viewer server (`src/viewer/server.ts`) gains two **additive, read-only** routes, each a fresh disk read with `Content-Type: application/json` and `Cache-Control: no-store`, mirroring the existing `GET /graph.json` pattern: (a) `GET /link.json` → the current dir's `.sutra/link.json` (404 with a JSON `{ error }` body when absent), and (b) a route that serves another linked repo's `graph.json` by the path/name recorded in `LinkResult.repos` (e.g. `GET /repo-graph?path=<encoded path>`), with a strict allowlist limited to the paths present in `link.json` (no arbitrary filesystem read, no `..` traversal — consistent with 3.1's path-traversal guard). The server still binds `127.0.0.1` only and imports none of `scanner.ts`/`checks.ts`/`features.ts` (3.1 leaf rule preserved).
2. The SPA adds an **Ecosystem** view (a new top-level tab/route alongside the 3.2 feature-cards grid in the `viewer/` SPA). It is reachable only when `GET /link.json` returns a valid `LinkResult`; when `link.json` is absent, the tab is hidden or disabled and shows an honest empty state ("Run `sutra link <repoA> <repoB>` to build the ecosystem map"), never a blank screen or stack trace (consistent with 3.1 AC5 graceful-absence handling).
3. Each repo recorded in the `LinkResult` renders as a visually bounded **cluster**, labelled by its repo name (and `commit`, if the `LinkResult` carries it). The cluster set is exactly the repos the linker recorded — no inferred or invented repos. The SPA validates the served `link.json` against `LINK_VERSION` (as 3.1 validates `graph.json` against `GRAPH_VERSION`) and shows a clear version-mismatch message rather than rendering stale data.
4. Cross-repo links are drawn from the `LinkResult`'s `LinkedEdge[]`. Each `LinkedEdge` carries repo-namespaced `from`/`to` ids in the `makeCrossRepoId` `<repo>::<relPath#symbol>` form; the link's source and destination clusters are derived by splitting those ids on the `::` separator (centralised by 1.4 in `src/util/ids.ts`), never guessed from labels.
5. **Confirmed-vs-candidate honesty (the core honesty gate):** a `LinkedEdge` with `resolution: "confirmed"` renders as a solid link; `resolution: "broken"` renders visibly distinct (e.g. red/dashed) and labelled "broken"; `resolution: "unresolved"` renders dimmed/dashed, labelled "unresolved", and is hidden behind a toggle that is **off by default**. No `broken` or `unresolved` link may be rendered with the same visual treatment (class/marker/color) as a `confirmed` one. (Regression-guarded in the Test Plan.)
6. Each cross-repo link is annotated with its **reconciliation status** sourced from Story 2.2. For a link whose destination is repo R, the SPA fetches R's `graph.json` (via the AC1 per-repo route) and finds the `SutraReconItem` in `R.reconciliation` whose `method` + `path` (and, where available, `handler_node`) match the link; its `status` (`matched` | `orphaned` | `dead` | `unresolved`) and `confirmed` boolean are shown on the link as an accessible text label plus a status badge/color. When no `SutraReconItem` matches (or the destination repo's graph is not retrievable), the link still renders from the 1.4 `resolution` alone and the reconciliation badge reads "n/a" — never fabricated.
7. The ecosystem graph renders at **cluster + cross-repo-endpoint granularity**, not every raw node: a cluster shows only the endpoints/handlers that participate in a cross-repo link plus a count of the repo's other nodes, so the map stays legible at product scale (echo-ai 638 / brain-api 1,218 nodes must NOT all be drawn).
8. The view is read-only and local-first: it adds no network calls beyond the localhost `GET /graph.json`, `GET /link.json`, and the per-repo graph route on the 3.1 server. No write-back, no auth, no `user_id`, no Brain-runtime dependency (BRIEF.md hard constraints, ROADMAP principle 6).
9. Any AI-derived field that might appear on a node/cluster label (e.g. a Story 2.3 AI feature name, if present) is rendered with an explicit AI marker and is never used to create, promote, or re-route a link — links come only from deterministic 1.4 `LinkedEdge`s plus 2.2 status (ROADMAP principle 1). The static `sutra view` (`src/view.ts`) renderer and the single-repo viewer behavior are unchanged; the Ecosystem view is purely additive. No `GRAPH_VERSION` or `LINK_VERSION` bump is introduced by this story — it is a consumer of existing schemas (ROADMAP principle 4).

## Technical Approach
This is a viewer-layer feature: a small additive extension to the 3.1 `src/viewer/server.ts` plus new SPA assets under `viewer/`. No scanner, checks, features, or graph-contract changes.

**Server (extend, do not replace, 3.1's `src/viewer/server.ts`):**
- Add `GET /link.json` → `fs.readFileSync(path.join(cwd, SUTRA_DIR, "link.json"))` per request, `no-store`, 404-with-`{error}` when absent (mirror the existing `GET /graph.json` handler exactly). Add `"link.json"` as a `LINK_FILE` constant in `src/types.ts` if 1.4 has not already (1.4 owns `LINK_VERSION`; the filename constant belongs next to `GRAPH_FILE`/`VIEW_FILE`).
- Add a per-repo graph route (e.g. `GET /repo-graph?path=<encoded>`): first read `.sutra/link.json`, build the set of allowed repo paths from `LinkResult.repos`, and serve `<thatPath>/.sutra/graph.json` only if the requested path is in that allowlist; otherwise 400/404. This is the single point that lets the SPA read a *second* repo's graph for the reconciliation join (AC6) without opening arbitrary filesystem reads.
- Keep the `127.0.0.1` bind and the leaf rule (no `scanner`/`checks`/`features` imports).

**SPA (new view under `viewer/`, in the 3.1 zero-build idiom):**
- Add an `viewer/ecosystem.js` (vanilla DOM module, same idiom as 3.1's `viewer/app.js`) and the markup/nav entry to switch into the Ecosystem view. Reuse the shared CSS (`viewer/styles.css`) and the pure render helpers in `src/viewer/render-shared.ts` (Mermaid helpers `buildMermaid`/`mermaidShape`/`mermaidLabel`/`esc`) so cluster rendering uses Mermaid `subgraph` blocks consistent with the static `src/view.ts` output — one source of truth, no drift.
- Build the pure ecosystem model in the SPA: clusters = `LinkResult.repos`; links = `LinkResult` `LinkedEdge[]`, with source/destination clusters obtained by splitting `from`/`to` on `::`. Carry `resolution`, `method`, `path` to each rendered link.
- Reconciliation join (AC6): for each link, fetch the destination repo's graph via the AC1 route, then match a `SutraReconItem` on `method`+`path` (+`handler_node` when present) and attach `status`+`confirmed`; default to "n/a" on no match.

**Types — reuse, do not redeclare.** Import `SutraGraph`, `SutraReconciliation`, `SutraReconItem`, `ReconStatus` (from 2.2), and `LinkResult`, `LinkedEdge`, `LINK_VERSION` (from 1.4) from the single `src/types.ts` contract. The zero-build SPA reads these as plain JSON; keep any TS-side helpers importing the canonical types so a contract change is a compile error, not silent drift.

**Honesty rules respected:** `confirmed`/`broken`/`unresolved` from 1.4 are preserved end to end and never collapsed; `unresolved` links default-hidden behind a toggle; reconciliation `confirmed: false` is surfaced honestly; AI fields (if any) are labelled and never mint or re-route links; all ids stay the deterministic `makeCrossRepoId` strings so the map is stable across re-scans.

**No contract changes / version bumps.** If the executor finds that 1.4's `LinkedEdge` lacks a field needed to join to 2.2's `SutraReconItem` (e.g. `method`/`path` cannot be recovered from `to`), that is a 1.4/2.2 gap — STOP and surface to the planner rather than adding fields to the schema here.

## Test Plan
Fixtures: reuse the Story 1.4 `tests/fixtures/xrepo-client/` + `xrepo-server/` pair, and the `LinkResult` that `linkGraphs([clientGraph, serverGraph])` produces, plus each repo's `SutraGraph.reconciliation` from Story 2.2's `reconcile`. Add a committed two-repo fixture for the SPA/server tests under `tests/fixtures/ecosystem/` containing repo-A's `.sutra/graph.json` + `.sutra/link.json` (a `LinkResult` with one `confirmed`, one `broken`, one `unresolved` `LinkedEdge`) and repo-B's `.sutra/graph.json` (carrying a `reconciliation` field). The server tests run `startViewerServer` against this dir on an ephemeral port (`port: 0`) and assert over HTTP with `fetch`, then `close()` — no browser, mirroring 3.1's Section-10 approach.

Describe blocks (continue the `tests/sutra.test.ts` Section convention after 2.2's Section 19 / 1.4's Sections 10–14):
- **Ecosystem server routes** — `GET /link.json` returns 200 + `no-store` + the fixture `LinkResult`; absent `link.json` → 404 with JSON `{error}`. The per-repo graph route serves repo-B's graph for an allowlisted path and rejects (400/404) a path not in `LinkResult.repos` and any `..` traversal attempt.
- **Ecosystem: clustering** — the model builds exactly two clusters from the two-repo `LinkResult`; the `confirmed` `LinkedEdge` yields one link whose source/destination clusters are correctly derived by splitting the `makeCrossRepoId` ids on `::`.
- **Ecosystem: confirmed vs candidate honesty** — the `confirmed` link renders solid; the `broken` link renders with a distinct (non-confirmed) treatment + label; the `unresolved` link is hidden by default and dashed/labelled when toggled on. **Regression guard:** assert no `broken`/`unresolved` link shares the confirmed link's rendered class/marker.
- **Ecosystem: reconciliation status** — a link joined to a `matched` `SutraReconItem` shows `matched` + `confirmed: true`; a link with no reconciliation match (or unfetchable destination graph) shows "n/a"; both appear as accessible text, not color-only.
- **Ecosystem: empty / version-mismatch state** — with no `link.json`, the Ecosystem tab is hidden/disabled with the honest empty state; a `link.json` whose `version !== LINK_VERSION` shows the mismatch message, not stale data (no crash).
- **Ecosystem: legibility** — a cluster whose graph has many nodes renders only the cross-repo-participating endpoints plus an "other nodes" count, not every node.
- **Regression guard: single-repo + leaf unaffected** — loading a single repo with no `link.json` leaves the 3.1/3.2 feature-cards view fully functional and the Ecosystem tab hidden; the new routes write nothing under `.sutra/`; `src/view.ts:renderView` output is byte-unchanged.

## Out of Scope
- **Producing cross-repo edges** — `linkGraphs`/`LinkResult`/`.sutra/link.json`/`makeCrossRepoId` are all Story 1.4. 3.4 only renders the edges 1.4 emitted.
- **Computing reconciliation status** — `reconcile`/`SutraReconciliation`/the `dead_endpoint` issue are Story 2.2. 3.4 only displays the `status`/`confirmed` it attached.
- **The viewer shell itself** — `sutra viewer`, `startViewerServer`, the SPA scaffold, and `render-shared.ts` are Story 3.1. This story only *extends* the server with two read-only routes and *adds* one view inside the existing shell.
- **Drill-down into a cluster's internal nodes** (node-level navigation into a repo's internals from the ecosystem map) — interplay with 3.2/3.3; 3.4 stops at cluster + cross-repo-endpoint granularity.
- **Auto-discovery of ecosystem membership** — the repos come from `LinkResult.repos` (what the user passed to `sutra link`); no registry/manifest of "which repos form a product" is built here.
- **Live / watch refresh** of the ecosystem map on file change — Story 3.5 (this story requires only a manual reload, like 3.1).
- **Search / filter / export** of the ecosystem view — Story 3.6.
- **Promoting candidate (`broken`/`unresolved`) links to confirmed, or any AI inference about cross-repo links** — 3.4 must not mint or upgrade links.
- **Non-HTTP cross-repo wiring** (events, queues) and **non-TS/JS repos** (Epic 4) — HTTP `LinkedEdge`s only.
